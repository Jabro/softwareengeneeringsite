Patterns

  1. Name and explain three Object Oriented Principles that are important in code design and thereby also in the use of patterns.

    inheritance: Is-a relationship. (abstract) superclasses hold data and methods. These methods are equal to all specific implementations.
    polymorphism: Use of methods or classes that specific shape is not known by the caller. But with well defined method signature or class inheritance, it is possible to use specific implementations.
    encapsulation: Hiding data of the specific implementation by restricting the access. Accessors can access the data via predifined methods. These methods are defined by the class/object holding the data.
    abstraction: Defining abstraction in classes, methods, types, interfaces and functionality reduces complexity.

  2. What is the difference between a Has-A and an Is-A relationship in various patterns. Name examples from a pattern that you know.

    Is-a: Inheritance. The (abstract) parent class can only exist if there is a specific implementation. With A Building only exists if I create a House (that extends the building). A ski only exists if I create a Telemark, Snowboard or Carving ski.
    Has-a: Relationship. The type that holds a bunch of other objects, can hold 0, 1 or much more objects. The existence of the object does not depend on the object it holds. 1-200 people can enter a train. But the trains does not disappear if there is no person in the train.

  3. How many of these explanations can you match to a pattern that you know about: 1. Encapsulate interchangeable behaviors and use delegation to decide which behavior to use 2. Subclasses decide how to implement steps in an algorithm 3. Subclasses decide which concrete classes to create

    1. Describes the strategy pattern: A class can select the a behaviour by selecting a specific implementation (... a specific strategy).
    2. Describes the template pattern: Abstract superclass defines the steps of an algorithm. The class represents every step with an abstract method. The subclass just has to implement the behaviour of an algorithm step
    3. Describes the factory pattern: Object creation with method instead of Constructor. The caller does not need to know the specific implementation, just gets it returned through the abstract class/interface. See abstract factory for more info.

  4. What is the difference between (1) Interface and (2) Abstract and (3) "normal" concrete classes?

    Interface: method signature definition to implement
    Abstract class: Generalize behaviour and data in a class that can not be instanciated. Classes can extend the abstract class and they can (if they are not abstract) be instanciated. Can implement an Interface. Can extend abstract classes.
    Normal class: Can be instanciated. Can implement an Interface. Can inherit from an interface, an abstract class and a normal class.

  5. What is the idea behind the Singleton Pattern. How does it do that?

    Is used if you only want to have one concrete instance of a class. It does that by providing a static access method to a private static class attribute of the class itself. You can only access the concrete instance by calling the method, wich will return you the one instance, hold by the class in a static context.

  6. You may or may not have prepared the Adapter pattern. Either way, if you hear the name, what could it possibly do and how could you go about separating the issues in two classes with an adapater class?

    Can "wrap" data structures. It "plugs" and can wrap a data structure from A to B for example.

  7. Describe a pattern of your choice. Since you cannot draw the diagram here, explain it in words (so that someone else could draw it). Explain what it does ie. when to use it. Gang of Four Design Pattern always has a part called participants, where their function is explained.

    Factory: Deals with object creation. It is a creational pattern. Define an interface for creating an object. Let the the classes that implement the interface decide which class to instantiate. The Factory method lets a class defer instantiation to subclasses.

Configuration Management

  1. What are the key differences between SVN and GIT? What are the pros and cons for using either.

    Key difference:
    GIT is decentralized system
    SVN is centralized.
    Git is a centralized system, which means every developer has it's own repo. Commit everywhere everytime. Decentralized version control systems are more complex (example: 2 ways to "commit": local commit (git commit) and "remote commit" (git push)). Often used in open source world with many different projects: everyone is free to pick interesting changes from other projects. With the 'staging' it is also possible to simply commit a single file or line that has change (also if you are in a huge "chaotic" change). Easy fix of mistakes before you push, because you can manipulate your commits ahead as you want.
    SVN is a decentralized system, which means that you cannot commit changes if you have no internet connection. The code is stored on the server only, you do not have a local repository.

  2. What is a change request form? What does it necessarily have to contain? Why is it used? What does a change request process look like in your project? (Fig. 13-14 in Bruegge book or Page 10 on the Configuration Management Slides - this is the picture of the workflow that we discussed and criticized in lecture) - what is missing in this diagram?

    A change request form contians information about the change that you want to implement. The control board can then decide what to do with the request: Either the board rejects it or it approves the request. If it is approved, the reqeust is assigned to a person (developer). He will then implement it, commit it via version control. Finally the commit representing a feature gets controlled by the quality control team.
    A change request form contains different fields with different informations. These information may contain the advantage and the overvalue that the feature creates for the customer. Anyhow, it contains information about how the software behaviour should be changed.
    The Project Manager of the board decides weather the request is approved or declined. The board and the Account Manager (which have customer contact) communicate with the customer about the current change requests and their corresponding requirements.
    Ideally, all this is automated and emails are generated with status change of change request.

Lifecycle Management

  1. What is lifecycle management?

    Continuous process during the project. Covers the general aspects of controling/governance, development and maintenance. (It contians the technical and organisational aspects, not the "soft skill" aspects like strategy, culture and leadership). Like business management controls their company, alm controls the continuous development of software. Lifecycle management is the generic term for tools/processes/management for the important aspects of software development: version control, issue tracking, test/model driven development, build management, software quality, requirements, project planning.
